The App
From building for yourself to building for everyone else
Claude Code Is Not a Chatbot
You've been using Claude Code as a place to do work. It's actually a place that runs work.
Tools
Read and write files. Execute bash commands. Query databases. Fetch APIs. Spawn sub-agents. These are hands, not suggestions.
Memory
CLAUDE.md at every level of the directory tree. Firm context, data maps, guardrails — accumulated and always active.
Autonomy
Multi-step workflows without checking in. Reads data, reasons over it, produces deliverables, writes them to disk. No human in the loop once it's running.
That combination — tools plus memory plus autonomy — is a different category of thing than a smart assistant.
The Harness
The execution environment that wraps Claude's intelligence with capability
CLAUDE.md
Persistent memory. Every file you've set up — firm overview, data schema, guardrails, workflow instructions — is active in every session. CC knows where it is and what it's working with before you type the first prompt.
Tools
The harness gives Claude hands: read a Supabase table, write an HTML file, run a bash command, call an API, spawn a parallel agent. Capability the model alone doesn't have.
Autonomy
Chain those steps without interruption. Query → reason → write → verify. The harness manages the execution loop. You define the goal; it runs the workflow.
The harness is the software. Claude is the reasoning engine inside it.

None of that required an app. It required an expert who knows how to drive the harness.

That's the ceiling for people who know how to use it. It's high.

The question isn't whether the harness can do it.
It's who can drive it — and what happens when they can't.
The Limit
The constraint is access and expertise, not capability
Access
Sarah doesn't use Claude Code. The analysts you hand deliverables to, the portco ESG contacts you report through, the clients reviewing your work — none of them are in the terminal. Your output shouldn't require CC to use it.
Expertise
Open-ended work requires domain judgment to scope. "Build me a carbon dashboard" means something different depending on who's driving. Getting better at prompts doesn't solve this — the bottleneck is knowing what to ask for in the first place.
The harness is where you work. The app is what you leave behind.

Design Question First
Before you build anything, answer one question: what do you want Sarah to be able to do?
Who is Sarah?
Associate, analyst, LP, portco ESG contact. Role determines what she needs to see — and what the app should expose.
What's the workflow?
What does she input? What does she get? What decision does it support? One workflow per app — scope creep is a design failure.
What does "done" look like?
She closes the app. What just happened? What can she do now that she couldn't before? That's your spec.
The app delivers exactly that. Everything else is out of scope.

    How to Run It
    No npm
    No build step. No node_modules. Vanilla JS with ES modules — works natively in any modern browser.
    One command
    python3 -m http.server 8080 — pre-installed on Mac. Opens at localhost:8080.
    Why a server?
    ES module imports require HTTP, not file://. The server is one line — it's not infrastructure.
    The API Call
    Direct to Claude
    fetch() to api.anthropic.com from agent.js. Your API key lives in the file — localhost only.
    Key safety
    Do not commit this file to git. CC will add a comment reminding you. Local demo only.
    Same key as CC
    The API key in your ~/.claude config. You already have it.
