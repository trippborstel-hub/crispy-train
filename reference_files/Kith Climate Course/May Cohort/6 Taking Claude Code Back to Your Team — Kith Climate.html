Taking Claude Code Back to Your Team

An overview of team working, version management and rolling Claude Code out inside a real organisation.

1Why this is the real frontier

Over six weeks you learned to build a system for yourself. The harder problem is building one for a group — and then for a group inside a large, often non-AI-native organisation.

2How it's managed at Kith

Our setup is deliberately simple — one shared workspace the whole team builds inside, with a few layers on top that let us move fast without tripping over each other:

  • One core SharePoint, synced to everyone. A single shared workspace sits at the centre. It syncs automatically to every team member's machine, so there's one source of truth — not five copies drifting apart on five laptops.
  • Folders wired to GitHub, GitHub wired to live URLs. Specific folders within that workspace are connected to GitHub repositories, which push to live URLs. You build in the folder, commit, and the change goes live — the same pattern you used throughout this cohort.
  • A tech team layered over the top. A small technical team maintains the plumbing — the repositories, the deploy pipelines, the connections — so everyone else focuses on the work rather than the infrastructure.
  • Whole-team access across the system. By default the team can reach the whole workspace. Openness is the starting point; restrictions are applied deliberately, only where they're needed.

3Restrictions and permissioning

Open by default doesn't mean open to everything. Access is shaped at two levels that work together:

  • Folder-level permissions in your file store. SharePoint, Google Drive or similar let you set permissions per folder — restricting who can edit, or even who can view, particular documents. This is the first and simplest control: sensitive material lives in folders only the right people can reach.
  • Account-level restrictions in Claude Code. Within an enterprise structure, Claude Code itself can be restricted at the individual-account level — governing what a given person's Claude is allowed to edit, not just what they can see in the file store. The file store controls access; Claude Code controls what actions are permitted on top of it.
  • Organisation-wide capability settings. At the enterprise tier there are also org-level controls worth scoping early — which models and features are enabled, what data can leave the environment, and what guardrails apply to every user by default. These set the floor everyone operates within, so they're a conversation to have with your IT and admin team up front, not after.

4Rollout tips

A few things that make the difference between a roll-out that sticks and one that stalls:

  • Start with a pilot. Don't switch everyone on at once. Pick one team and one workflow, prove the value, then widen from there.
  • Start with a team that all share the same access. Beginning with a group who have identical permissions removes a whole layer of friction — nobody's blocked, everyone's working from the same files — so you can focus on the work rather than untangling who-can-see-what. Add access complexity later, once the pattern is proven.
  • Bake controls and versioning into CLAUDE.md and reference files. Don't leave the rules in people's heads. Encode your controls function and versioning conventions directly into the CLAUDE.md and reference files, so the discipline travels with the work and every team member's Claude follows the same playbook.
