# crispy-train — EY Climate AI Agent Training Build

## What this project is

We are building a training program to teach EY Climate practice colleagues how to use
AI coding agents (Cursor + Claude/GPT today; Claude Code possibly later). The program is
modeled on the Kith Climate 6-week cohort, whose materials live in `reference_files/`.

## Who is working here

- **Tripp** (Managing Director, EY Climate) — project lead, Kith Climate graduate
- **Miranda Nayyar** — co-instructor, Kith Climate graduate
- **Dan Kenney** — co-instructor

All three collaborate asynchronously on this repo from personal machines. No EY client
or proprietary data may ever be committed here — the training uses a fictional client
sandbox (Riverbend Beverage Group) instead.

## Current state

See `course-build/project-brief.md` (charter), `course-build/pilot-arc-v1.md` (the
4-week program design), and the most recent files in `sessions/` before proposing

- `index.html` — student-facing course hub (front page of the published site)

  never edit, never reuse in student-facing materials)

Related repo: `the-dyad-and-the-deliverable` (standalone public essay synthesizing the
Kith lectures; deliberately separate from this course).

## Guiding principles (agreed by Tripp; do not violate)

1. **Publication awareness.** This repo is PUBLIC and served live via GitHub Pages at
   `https://trippborstel-hub.github.io/crispy-train/`. Every commit publishes to the
   open web. Before committing anything sensitive, pause and ask. A split into a
   private instructor repo + public student repo is planned; until then, treat every
   file as world-readable.
2. **Ethics & IP framework** (full version: `course-build/source-map-v1.md`). The Kith
   course is our inspiration and ONE source among several — never the sole source for
   any module. Ideas are commons; Kith's *expression* (artifacts, decks, Northwood,
   coinages, sequencing) is their IP: never copy it into student-facing materials.
   Three tests govern everything we build: **transformation** (our audience, our
   harness, our artifacts), **attribution** (Kith credited in Week 1, hub footer, and
   instructor materials; named coinages credited to Diego/Kith), **permission**
   (Diego's blessing pending — internal training is aligned with Kith's mission;
   client-sold packaging of Kith frameworks is out of bounds, permanently).
3. **Source ledger.** When you build or substantially revise a module, note its sources
   against `course-build/source-map-v1.md` and add new sources there. A module leaning
   on a single source is a defect: diversify or attribute prominently.
4. **Concepts over harness.** Teach agent concepts with Cursor as the vehicle, so the
   skills survive a harness change. Name the concepts (agent, harness, context, rules
   files, reference files) rather than teaching buttons.
5. **Fictional data only.** No EY client data, no real client names, no EY-internal
   material, no real credentials anywhere in this repo. Riverbend is the sandbox; its
   data is generated by `course-build/tools/`.
6. **`[EY-NETWORK]` convention.** Anything requiring EY-internal specifics (IT,
   procurement, policy, Risk Management, Teams links) is marked `[EY-NETWORK — Tripp]`
   in the files rather than guessed at. Only Tripp resolves these.
7. **Kith's infrastructure is off-limits.** Their Supabase environment and API keys
   (in the local-only northwood folder, which stays untracked) are never used,
   committed, or distributed. `.gitignore` blocks `*api-keys.md` as a backstop.

## Conventions for every AI agent working in this repo

1. **End-of-session log.** Before a working session ends, write a summary to
   `sessions/YYYY-MM-DD-<topic>.md`: decisions made, rationale, artifacts created/changed,
   open questions, next steps. Write it for a colleague who wasn't there.
2. **Catch up before starting.** At the start of a session, read the two or three most
   recent files in `sessions/` so you know what your collaborators decided.
3. **Versioning.** Drafts are files with `-v1`, `-v2` suffixes. Never overwrite a version
   another collaborator may be reviewing; add a new one.
4. **Commit and push at the end of every session** so the others can see the work
   (remembering principle 1: pushing = publishing). Commit messages say what changed
   and why in one line.
5. **Instructor-only materials** (answer key, reference totals, generator scripts,
   source map) live in `course-build/` and are never copied into `sandbox-client/` or
   any future student-facing repo.
